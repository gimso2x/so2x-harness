# Spec Lite: v0.1.1 Solid Kit

## Goal

so2x-harness v0.1의 완성도를 autopus-adk, hoyeon 수준으로 끌어올리기.
Shell+Python 정체성을 유지하면서 품질, 테스트, CI/CD, 문서, 스킬/hook 확장을 수행한다.

## Key Decisions

1. **Shell+Python 유지** - Go/Node 마이그레이션 없이 현재 기술 스택에서 품질 향상
2. **품질 우선, 기능 나중** - autopus/hoyeon의 완성도를 먼저 매칭하고, 복잡한 기능은 v0.2+에서
3. **pytest 기반** - 가벼운 키트에 맞는 표준 테스트 프레임워크
4. **GitHub Actions** - CI/CD를 위한 표준 도구
5. **점진적 스킬/hook 확장** - 검증된 패턴만 4-6개 추가

## Requirements

### R1: 테스트 인프라
- 모든 lib 모듈에 단위 테스트 작성
- apply/update/doctor에 통합 테스트 작성
- lib 모듈 커버리지 80%+, 스크립트 커버리지 60%+
- pytest + tmp_path로 격리된 환경에서 실행

### R2: CI/CD
- GitHub Actions CI 워크플로우 (push/PR 시)
- Python 3.10/3.11/3.12 매트릭스 테스트
- 린트, 타입체크, VERSION/harness.yaml 검증
- 릴리즈 자동화 워크플로우

### R3: 스킬 확장 (6 -> 10-12)
- spec-validate: spec-lite 문서 구조/완결성 검증
- setup-context: 프로젝트 분석 -> 컨텍스트 문서 자동 생성
- changelog: 변경 이력 구조화 기록
- safe-commit: 커밋 전 검증 (시크릿, 파일 크기, 범위)

### R4: Hook 시스템 확장 (1 -> 4-5)
- pre-apply-check: apply 전 대상 프로젝트 상태 검증
- post-apply-verify: apply 후 설치 결과 검증
- pre-commit-check: 커밋 전 시크릿/범위/검증 상태 확인
- spec-gate: spec-lite 사용 시 완결성 게이트

### R5: 문서 보강
- ARCHITECTURE.md: 시스템 구조, 모듈 책임, 데이터 흐름
- CONTRIBUTING.md: 기여 가이드, 개발 환경, PR 프로세스
- CHANGELOG.md: 버전별 변경 이력
- docs/cli-reference.md: 전체 명령 참조
- docs/preset-guide.md: 프리셋 작성 가이드

### R6: 스크립트 품질 개선
- 일관된 에러 코드와 메시지
- 경로/플랫폼/프리셋 파라미터 검증
- update 전 백업, 실패 시 복원
- config.json, manifest.json JSON 스키마 정의
- Makefile: test, lint, coverage, doctor

## Risks / Unknowns

- autopus/hoyeon은 Go/Node 생태계라 Python에서 동등한 품질을 달성하는 방법이 다를 수 있음
- 스킬/hook 추가 시 기존 설치된 프로젝트와의 호환성
- CI 매트릭스 테스트 시간이 길어질 가능성

## Implementation Steps

1. Makefile 작성 (test, lint, coverage, doctor)
2. 테스트 인프라 구축 (conftest, fixtures)
3. lib 모듈 단위 테스트 작성
4. 스크립트 통합 테스트 작성
5. CI 워크플로우 작성
6. 스크립트 에러 처리/입력 검증 개선
7. JSON 스키마 정의
8. 백업/롤백 메커니즘 추가
9. 신규 스킬 템플릿 작성
10. Hook 템플릿 작성
11. 문서 작성 (ARCHITECTURE, CONTRIBUTING, CHANGELOG, CLI reference, preset guide)
12. 릴리즈 워크플로우 작성
